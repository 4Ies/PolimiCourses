How can we define what is a secure system? 
Usually, it follows the \textbf{CIA} paradigm to correctly define what a 
secure system is, the problem lies inside its definition:
\begin{itemize}
    \item \textbf{Confidentiality} - the system should not leak information to 
        unauthorized parties.
    \item \textbf{Integrity} - the system should not allow unauthorized parties 
        to modify data \textbf{and} should not allow authorized parties 
        to modify data in an unauthorized way.
    \item \textbf{Availability} - information should be always available 
        to authorized parties.
\end{itemize}
The problem lies inside \textbf{Integrity}, if deciding whether a party is
authorized or not is simple since it's a binary decision, the decision of
whether a modification is authorized or not is much more complex and involves
a lot of factors. Also, \textbf{Availability} is what pays off in the end.
What can we consider \textbf{secure}? Something is secured if it takes
an \textbf{effort} to break it: the more effort it takes, the more secure 
it is.
\begin{key}{Secure System}
    A \textbf{Secure System} is a system that requires a lot of effort to be 
    broken.
\end{key}
\subsection{Vulnerabilities and Exploits}
We firslty need some definitions useful to understand the world of security:
\begin{itemize}
    \item \textbf{Vulnerability} - It is a shortcut that allows an attacker
        to break the system with less of the intended effort: it is something 
        that allows to violate one of the CIA properties.
    \item \textbf{Exploit} - It is a specific way to use one or more
        vulnerabilities to accomplish a specific objective that violates
        one of the CIA properties.
\end{itemize}
\begin{warning}{Understanding what is and what isn't a vulnerability}
    Not all vulnerabilities lookalikes are actual vulnerabilities or mistakes
    to exploit: sometimes, something is vulnerable but the accomplishment of 
    the attack doesn't result in a violation of the CIA properties.  
\end{warning}
\subsubsection{Example: exploiting a vulnerability}
When exploiting a vulnerability, we firstly need to find the security 
assumption that lays behind the system: in this example, the thing that 
ensures that the lock works is the fact that the pins are being pushed down 
always by the strings of the lock, so the only way to open the lock is to 
push up all the pins at the same time. Lockpicking is using a vulnerability 
of the lock to open it: this is using an \textbf{exploit}, using something 
to take advantage of a vulnerability to break the system.

    \caption{Lockpicking example}
\end{figure}

How can we resolve this vulnerability? Short answer: we make it harder to 
exploit it. Oiled locks are harder to exploit because of the deletion of 
friction, but still are vulnerable: some vulnerabilities are implicit in 
the design of the system itself.
\subsubsection{Example: A strawman software vulnerability}
\begin{code}{Strawman software vulnerability}
    \begin{verbatim}
    int i;
    unsigned short s;
    i = atoi(argv[1]); // parse command line parameter as int
    if (i == 0) { // check
        printf("Invalid number: value must be > 0\n");
        return -1;
    }
    s = i;
    if (s == 0) { //security check
        printf("Access GRANTED!\n");
    }
    \end{verbatim}
\end{code}
Implicit casting from int to unsigned short can cause a vulnerability (in 
this case, to have 0 in the variable \textbf{s} even if \textbf{i = 65536}).

\subsection{Assets, Threats, Attacks and Threat Agents}
\subsubsection{What is Protected and what is Secured}
It is crucially important that it is different to say that something is 
\textbf{safe} and that something is \textbf{protected}: something can be 
more protected but less safe, because of the presence of specific 
\textbf{threats} or because of its value for the individual threat (a 
soldier is less protected but safer against a terrorist than the polititian 
inside the ACV). The Cheyenne Mountain is protected against nuclear attacks,
but it is not safe against rattlesnakes.
\begin{key}{Safety agains Threats}
    A system is protected (and safe) always against a specific threat,
    it is of no sense to talk about the security of a system without
    considering the threat we are trying to protect against.
\end{key}
\begin{warning}{Watch Out!}
    It is total bullshit to talk about protecting a system and potential 
    vulnerabilities without having a clear understanding of the threat models: 
    professionals in the field of security only talk about them, 
    not about vulnerabilities.
\end{warning}

\subsubsection{Assets}
\begin{key}{Asset}
    An \textbf{asset} identifies what is valuable for the organization.
\end{key}
We will be focusing mostly on \textbf{IT Assets}, like data, hardware and 
software: identifying assets is key into value them from the point of view
of an attacker. Different attackers may have a different value for our 
different assets, it is our job to understand what could become the value 
of our resources and the potential loss of our assets.

\subsubsection{Threats and Attacks}
\begin{key}{Threat}
    A \textbf{threat} is a potential violation of a CIA rule and the 
    circumstances that can lead to said violation
\end{key}
A threat is usually caused by an intentional attacks, caused by attack agents
(we will not cover accidents, non intentional attacks).
\begin{warning}{Hakers are not Attackers}
    A Hacker is someone with an advanced understanding of computers and computer
    networks, malicious hackers are called \textbf{black hats} 
    (from western movies).
\end{warning}

\subsection{Risks}
We just understood that \textbf{no system is invulnerable}: there will always
 be a vulnerability that could become a threat to our system from a 
 specific point of view of an attacker. The only thing we can talk about is 
 \textbf{how exposed to damage} is our system: we are talking about 
 \textbf{risks}.
\begin{key}{Risk}
    A \textbf{risk} is a statistical and economical evaluation of the exposure
    to damage because of the presence of vulnerabilities and threats.
\end{key}
Risk can be defined as a mathematical formula:
\begin{equation}
    \text{Risk} = \text{Asset} \times \text{Vulnerabilities} \times 
    \text{Threats}
\end{equation}
A big problem is that the Risk term is almost always impossible to compute: we
hardly know the value of our assets from an attacker point of view, we have
little information on how vulnerable our system is and mostly important we are
never able to identify the threats our system will encounter. It is important
to note though that Assets and Vulnerabilities are 
\textbf{controllable variables}, the values we can actually manipulate, whereas
Threats are \textbf{indepentent variables}, completely separated from our 
knowledge. It becomes clearer and clearer that we need to take decisions:
we need to balance the urge of damage containment and the reduction of 
vulnerabilities to the costs they bring. This is \textbf{Security}.

\subsection{Security vs Cost Balance}
In security we can distinct two different costs:
\begin{itemize}
    \item \textbf{Direct Costs} - Management, operational and equipment
    \item \textbf{Indirect Costs (more important)} - Less usability, slower
    performance, less privacy, reduced productivity
\end{itemize}
(remember when no one locked their phones when biometrics was not a thing? 
it was more the cost of usability than the perceived threat of having the 
phone opened by someone else!)
\begin{warning}{The Privacy Tradeoff Problem}
    When the diminuition of privacy is traded with "more security", since 
    security is composed by privacy, it means something bad is going to happen. 
\end{warning}

\subsection{More money is not more security}
Sometimes a more sophisticated approach in trying to solve menaces brings
only indirect costs: in an airplane, liquids are not allowed over a 100ml 
capacity but batteries are allowed in almost every size or form, why? Because
a different approach limiting batteries would make all people avoid taking the
airplane! \newline
Lowering the security measures of a system usually doesn't happen because
someone has to decide to lower it: the blame of the system having a flaw will
eventually be his/hers.

\subsection{Trust}
It's hard to give a general definition of trust, in general or by referring
specifically to computer science.
In Computer Science, trust in software means that the code i wrote is "secure" 
(the logic itself is exactly the logic i had in mind) and also that the 
\textbf{compiled code} does exactly what i want to do: by securing the 
compiler layer of a piece of software that I wrote, I'm left with other 
variables that can be deemed to be external (the OS where it gets executed, 
the hardware that meets the piece of code that i wrote). Because of this 
complexity in defining what can be secured and what is to be trusted, 
we make \textbf{assumptions}.
\begin{key}{Trust Issues}
    In general security, in any case there will be a moment where my 
    assumptions cannot be validated further: this defines the 
    \textbf{trust boundary} of my system and assumptions.
\end{key}
The trust boundary depends on our maximum effort to verify if something is 
trustworthy or not (if we are a small business we will have a smaller trust 
boundary for an agent than the penthagon).
